后端服务层从零到成熟的 8 个步骤
1. 明确服务层边界

一句话概括：先定义后端服务层“负责什么、不负责什么”，并把 AssetService / BorrowService / ReturnService / InboundService / RuleService 的职责切开，避免后面代码职责混乱。

2. 统一服务层内部契约

一句话概括：在写实现前，先统一服务层内部传递的数据对象，至少明确请求对象、响应对象、硬件事件对象和 PendingTransaction 事务对象，保证 request_seq / hw_seq / asset_id / user_id / action_type 这些核心字段语义一致。

小步骤：

2.1 定义借出/归还/入库请求对象
2.2 定义统一服务响应对象
2.3 定义硬件事件对象
2.4 定义 PendingTransaction 结构
3. 先做规则层，再做流程层

一句话概括：先把“能不能借、能不能还、能不能入库”的规则集中到 RuleService，让所有非法请求都在进入硬件确认前被拦截。

小步骤：

3.1 借出规则校验
3.2 归还规则校验
3.3 入库规则校验
3.4 基础异常规则校验（黑名单、状态冲突、不可用等）

4. 设计事务状态机

一句话概括：把借出、归还、入库这类跨时间的长事务统一抽象成状态机，至少明确 IDLE → WAIT_ACK → WAIT_HW → UPDATING → DONE/FAILED 这条主链路。

小步骤：

4.1 定义事务状态
4.2 定义状态迁移条件
4.3 区分 ACK_TIMEOUT 和 HW_RESULT_TIMEOUT
4.4 定义失败收口规则

5. 先把数据库原子提交机制做成熟

一句话概括：在写完整业务流程前，先保证“资产状态更新 + operation_records 流水写入”必须作为一个原子事务提交，避免出现状态变了但流水没落库的半成功问题。

小步骤：

5.1 设计借出原子提交
5.2 设计归还原子提交
5.3 设计入库原子提交
5.4 定义失败回滚策略

6. 实现三条核心业务编排链路

一句话概括：在规则层、状态机、原子提交都确定后，再正式实现 BorrowService / ReturnService / InboundService 三条主流程，使其全部遵循“预检查 → 创建待确认事务 → 硬件确认 → 原子落库”的统一模式。

小步骤：

6.1 实现 BorrowService
6.2 实现 ReturnService
6.3 实现 InboundService
7. 接入硬件事件驱动闭环

一句话概括：把串口程序上报的 ACK、用户确认事件、心跳等输入转换成服务层可处理的业务事件，由事务管理器完成事务匹配、状态推进、幂等控制和超时收口。

小步骤：

7.1 接 ACK 事件
7.2 接 EVT_USER_ACTION 事件
7.3 做重复事件幂等处理
7.4 做离线与超时事务清理
8. 补齐成熟化能力

一句话概括：最后补齐日志、WebSocket 状态通知、统一错误码和测试体系，把“能跑”的服务层提升成“可联调、可排错、可演示”的成熟服务层。

小步骤：

8.1 建立业务/状态机/错误日志
8.2 建立 WebSocket 实时通知
8.3 统一错误码与错误响应
8.4 补齐单测与联调测试
