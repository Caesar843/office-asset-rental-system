# 后端服务层子目录约束

## 1. 当前阶段定位

当前目录的工作属于本项目第五步：在既有服务层边界、服务层内部契约、规则层契约与事务状态机设计不变的前提下，完成后端服务层的实现级拆分与编码落地。

本阶段不是重新定义需求，不是继续抽象设计，也不是推倒现有实现重写一套新系统。

---

## 2. 当前正式业务范围

当前 V1 正式范围仅包括以下两条主链路：

- Borrow（借出确认）
- Return（归还确认）

以下内容不属于当前目录本轮正式落地范围，不得擅自纳入主链路实现：

- Inbound（入库正式流程）
- 预约
- 续借
- 验收
- 盘点
- 多角色复杂权限体系
- 完整看板统计实现
- 完整 Web API 层
- WebSocket 完整推送体系

可以为后续扩展预留接口，但不得把上述内容硬塞入当前 V1 核心闭环。

---

## 3. 当前目录的职责边界

### 当前目录负责
- Borrow / Return 流程编排
- 调用规则层进行前置裁决
- 创建与推进待确认事务
- 发起硬件确认请求
- 接收事务推进结果
- 触发数据库原子提交
- 构建统一 BusinessResult
- 补充并维护当前 V1 后端核心测试

### 当前目录不负责
- 串口协议封包与拆包
- CRC 校验
- ACK 重传
- 心跳检测
- 底层串口连接管理
- SQL 底层细节
- 前端展示逻辑
- 视觉识别算法本体

---

## 4. 与串口通信层的关系

串口通信层在当前项目中承担协议、ACK、重传、心跳、去重与底层收发职责，当前已具备 V1 可运行实现。

当前目录的实现必须建立在该既有分层基础上，不得主动把协议处理、串口管理、ACK 重传、心跳检测等职责重新拉回后端服务层。

如确需为了后端服务层落地而调整与通信层的接口衔接，只允许进行最小必要修改，且不得破坏既有协议结构、ACK 语义、重传机制、心跳机制、去重机制以及 request_seq 与发送 seq_id 的对齐语义。

---

## 5. 必须保持的事务状态机语义

正式状态主链固定为：

- WAIT_ACK
- WAIT_HW
- UPDATING
- COMPLETED / FAILED

必须满足以下约束：

- 只有 ACK_OK 后才能进入 WAIT_HW
- 只有收到合法匹配且 `confirm_result == CONFIRMED` 的 EVT_USER_ACTION 后才能进入 UPDATING
- 只有原子提交成功后才能进入 COMPLETED
- 任一失败路径进入 FAILED
- 非法事件、重复事件、孤儿事件、迟到事件不得推进事务状态

不得擅自新增新的正式主状态并改写当前 V1 主链路语义。

---

## 6. 必须保持的硬件事件匹配规则

不得仅凭 `asset_id` 匹配硬件事件。

必须保持以下联合匹配规则：

- `asset_id`
- `request_seq`
- `action_type`
- `request_id`

若事务中已存在 `request_id`，则硬件回调必须带回该值，且必须完全一致。

以下事件必须直接忽略，不得推进事务：

- 缺少关键字段的事件
- `request_seq` 不匹配事件
- `action_type` 不匹配事件
- `request_id` 不匹配事件
- 重复事件
- 孤儿事件
- 迟到事件

---

## 7. 必须保持的持久化与追溯约束

当前业务流水必须能够关联硬件确认记录。

`operation_records` 必须保留并正确写入以下关键字段：

- `hw_seq`
- `hw_result`

必须保持以下原子提交边界：

- 资产状态更新
- 业务流水写入

两者必须同成功或同失败，不得出现以下半成功状态：

- 资产状态已更新，但流水未写入
- 流水已写入，但资产状态未更新

---

## 8. 当前正式资产状态范围

当前正式资产状态仅包括：

- IN_STOCK
- BORROWED
- MAINTENANCE
- SCRAPPED

不得擅自扩展新的正式资产状态并写入当前 V1 主链路。

---

## 9. 当前目录的本轮实现目标

当前目录本轮仅允许进行以下实现工作：

- 将前置规则裁决从集中式服务实现中抽离为独立规则层
- 将事务推进逻辑从集中式服务实现中抽离为独立事务状态控制层
- 将现有服务实现收敛为以流程编排为主的服务层
- 补充并完善测试
- 在不破坏既有通信层语义的前提下，完成 Borrow / Return 的 V1 后端核心闭环收口

本轮不允许偏离为以下方向：

- 重写整个项目
- 重新设计通信协议
- 把入库、预约、续借等非当前范围内容并入正式主链路
- 跳过测试直接堆实现
- 推翻现有核心对象体系另起一套命名体系

---

## 10. 对现有核心对象与字段语义的要求

当前实现必须建立在既有核心对象与字段语义之上，不得擅自引入与现有正式字段等价的新命名体系。

必须保持正式核心字段语义：

- `asset_id`
- `user_id`
- `action_type`
- `request_seq`
- `request_id`
- `hw_seq`
- `hw_result`

不得并行引入与其等价但命名不同的新主字段，如：

- `biz_seq`
- `tx_id`
- `confirm_seq`
- `asset_code`

---

## 11. 实现原则

- 必须基于现有代码增量演进
- 必须遵循既有服务层边界
- 必须遵循既有内部契约
- 必须遵循既有规则层契约
- 必须遵循既有事务状态机契约
- 必须优先在既有对象体系基础上完成显式化与拆分
- 必须保持当前可运行主链路语义不被破坏
- 必须输出可运行代码，禁止伪代码
- 必须补测试，不能只改实现不验证

---
